# Refactor de Materialización de Commands Globales

## Resumen

Se mantiene la arquitectura en cuatro piezas bien separadas:

- **Canónico**: `ai/assets/commands/registry.yaml` y `ai/assets/commands/<id>/COMMAND.md`
- **Adapters**: `ai/adapters/<platform>/...`
- **Derivados efímeros**: `build/commands/<platform>/<id>.md`
- **Runtime final**: materialización al HOME real vía `scripts/materialize-commands.sh`, invocable manualmente y desde Chezmoi

Se consolida la **Estrategia B** porque encaja mejor con Chezmoi:
- el repo no guarda artefactos derivados por plataforma en su árbol estable
- Chezmoi sigue siendo el orquestador natural del despliegue al HOME
- el build queda explícito, depurable y testeable sin mezclar “generate” con “publish”

## Cambios de implementación

### Scripts y flujo
- Refactorizar `scripts/generate-commands.sh` para generar únicamente en `build/commands/`.
- Crear `scripts/materialize-commands.sh` como script principal de publicación.
- El script de materialización debe:
  - usar `HOME` por defecto
  - aceptar override con `COMMANDS_HOME_ROOT`
  - permitir tests/debug materializando a un root temporal sin tocar el HOME real
  - generar o reutilizar `build/commands/` antes de sincronizar
  - sincronizar solo los archivos esperados por plataforma
  - dejar logs mínimos y claros
- Añadir `.chezmoiscripts/run_after_12_materialize_ai_commands.sh.tmpl` con lógica mínima:
  - invocar `scripts/materialize-commands.sh`
  - no renderizar ni duplicar reglas
  - fallar de forma visible si la materialización falla

### Política de materialización y limpieza
- Destinos reales gestionados:
  - `~/.config/opencode/commands/`
  - `~/.cursor/commands/`
  - `~/.codex/prompts/`
- La limpieza de obsoletos debe ser **precisa**:
  - solo actuar dentro de esas superficies gestionadas
  - no borrar directorios completos
  - no tocar archivos ajenos al sistema
- Política recomendada:
  - calcular la lista esperada desde `registry.yaml` + `build/commands/`
  - eliminar solo archivos `.md` gestionados por este sistema cuyo nombre corresponda a commands del registry pero que ya no deban existir en ese destino
  - no borrar otros `.md` manuales si no pertenecen a un command registrado
- Si hace falta para seguridad, el script puede tratar como “managed set” exclusivamente los IDs presentes en el registry.

### Validación y tests
- Ajustar `scripts/validate-commands-structure.sh` para separar:
  - validez estructural del sistema
  - existencia o no de artefactos generados
- Si `build/commands/` no existe:
  - no es error estructural
  - puede ser aviso informativo o sección omitida
- Actualizar tests para cubrir:
  - generación en `build/commands/...`
  - materialización con `COMMANDS_HOME_ROOT` apuntando a un directorio temporal
  - limpieza segura de obsoletos dentro de superficies gestionadas
  - mantenimiento del formato correcto en OpenCode, Cursor y Codex

## Flujo final

### Edición
- `ai/assets/commands/registry.yaml`
- `ai/assets/commands/<id>/COMMAND.md`

### Generación
- `./scripts/generate-commands.sh`
- salida en:
  - `build/commands/opencode/...`
  - `build/commands/cursor/...`
  - `build/commands/codex/...`

### Validación
- `./scripts/validate-commands-structure.sh`
- validación estructural independiente de si `build/commands/` existe

### Materialización
- manual:
  - `./scripts/materialize-commands.sh`
- manual con root alternativo para tests/debug:
  - `COMMANDS_HOME_ROOT=/tmp/commands-home ./scripts/materialize-commands.sh`
- vía Chezmoi:
  - `chezmoi apply`
  - el hook solo orquesta la llamada al mismo script

### Runtime final
- `~/.config/opencode/commands/<id>.md`
- `~/.cursor/commands/<id>.md`
- `~/.codex/prompts/<id>.md`

## Casos de prueba y aceptación

- `sos` se genera en `build/commands/opencode/sos.md`
- `sos` se genera en `build/commands/cursor/sos.md`
- `sos` se genera en `build/commands/codex/sos.md`
- `build/commands/` ausente no rompe la validación estructural
- materialización con `COMMANDS_HOME_ROOT` publica en un directorio temporal correcto
- materialización por defecto publica en el HOME real correcto
- si un command deja de estar habilitado para una plataforma, se elimina solo su archivo gestionado en esa plataforma
- no se borran archivos ajenos o manuales dentro de los directorios destino
- `chezmoi apply` deja el runtime consistente usando el mismo flujo que el modo manual

## Supuestos y defaults

- `COMMANDS_HOME_ROOT` será la variable de override para tests/debug
- `HOME` sigue siendo el default cuando no hay override
- `build/commands/` se ignora en Git
- el hook de Chezmoi se ejecuta después de los hooks actuales de runtime/links y solo actúa como orquestador
- la invocación funcional no cambia:
  - OpenCode: `/<command>`
  - Cursor: `/<command>`
  - Codex: `/prompts:<command>`
